## 1. Railway 배포 초기 오류 수정

### 문제

Spring Boot 백엔드를 Railway에 배포할 때 plain jar 충돌로 빌드 실패.

```
Error: more than one jar file found
```

Gradle 빌드 시 `app-plain.jar`와 `app.jar` 두 개가 생성되어 Railway가 실행할 jar를 특정하지 못하는 문제.

### 원인

Spring Boot Gradle 플러그인의 기본 동작으로 plain jar(의존성 미포함)와 executable jar(Fat jar) 두 개가 동시에 생성됨.

### 해결

`build.gradle`에서 plain jar 생성을 비활성화.

```groovy
jar {
    enabled = false
}
```

### 관련 커밋

- `fix: plain jar 생성 비활성화 (Railway 배포 오류 수정)`

## 2. Supabase ENOIDENTIFIER DB 연결 오류

### 문제

Railway 배포 후 백엔드 서버가 올라왔다 죽기를 반복. Supabase 연결 시 아래 오류 발생.

```
Caused by: org.postgresql.util.PSQLException:
FATAL: (ENOIDENTIFIER) no tenant identifier provided
```

### 원인 분석

Supabase 트랜잭션 풀러(포트 6543)는 멀티테넌트 구조로, PostgreSQL 연결 시 **`postgres.PROJECT_REF` 형식의 username**을 통해 테넌트를 식별함. 일반 `postgres` username만으로는 어느 프로젝트에 연결할지 알 수 없어 오류 발생.

**시도 1 (실패)** — HikariCP `data-source-properties`에 options 파라미터 주입

```yaml
spring:
  datasource:
    hikari:
      data-source-properties:
        options: project=rktozresjijfertgcqet
```

PostgreSQL 드라이버가 `options`를 시작 파라미터로 전달하나, Supabase 풀러가 이 방식을 인식하지 못함. 또한 이 블록 자체가 드라이버에 잘못된 파라미터를 보내 연결을 불안정하게 만드는 부작용도 발생.

**시도 2 (실패)** — 환경 변수 `DB_USERNAME=postgres.rktozresjijfertgcqet` Railway에 설정

Railway 환경 변수 파싱 과정에서 `.` 포함 문자열이 손상되는 것으로 추정되어 실제 적용이 안 됨.

### 해결

`data-source-properties` 블록을 완전히 제거하고, username을 `application.yaml`에 직접 하드코딩.

```yaml
spring:
  datasource:
    url: ${DB_URL}
    username: postgres.rktozresjijfertgcqet   # 환경변수 대신 하드코딩
    password: ${DB_PASSWORD}
    driver-class-name: org.postgresql.Driver
```

환경 변수 파싱을 우회해 안정적으로 테넌트 식별자를 전달. 이후 정상 배포 및 회원가입·로그인 모두 동작 확인.

> **참고**: Supabase Transaction Pooler 사용 시 username은 반드시 `postgres.{project_ref}` 형식이어야 하며, Session Pooler(포트 5432) 또는 Direct Connection(포트 5432)은 해당 없음.

### 관련 커밋

- `fix: Supabase pooler tenant identifier options 추가`
- `fix: Supabase pooler ENOIDENTIFIER 오류 수정`
- `fix: DB username 하드코딩으로 ENOIDENTIFIER 근본 원인 제거`

## 3. Expo SDK 버전 호환성 오류

### 문제

Expo Go 앱에서 연결 시 버전 불일치 오류 발생.

```
This project is incompatible with the installed version of Expo Go
```

### 원인

`package.json`의 Expo SDK 버전이 설치된 Expo Go 클라이언트와 맞지 않음.

### 해결

Expo SDK를 54로 업그레이드.

```bash
npx expo install expo@~54.0.0
```

의존 패키지들도 SDK 54 호환 버전으로 일괄 업데이트.

### 관련 커밋

- `fix: Expo SDK 54로 업그레이드`

## 4. 프론트엔드 API URL 미연결 문제

### 문제

로그인/회원가입 시 네트워크 요청이 실패하거나 응답이 없음.

### 원인

`frontend/src/constants/config.js`의 `BACKEND_URL`이 로컬호스트 또는 빈 값으로 되어 있어 Railway 배포 서버에 도달하지 못함.

### 해결

Railway에서 발급된 실제 배포 URL로 업데이트.

```js
// before
export const BACKEND_URL = 'http://localhost:8080';

// after
export const BACKEND_URL = 'https://easypack-back-production.up.railway.app';
export const AI_SERVER_URL = 'https://easypack-ai-production.up.railway.app';
```

### 관련 커밋

- `fix: 프론트엔드 API URL 실제 Railway 배포 URL로 업데이트`

## 6. 수하물 규정 데이터 최신화

### 배경

2024-2025년 기준으로 보조배터리 규정이 강화되는 등 항공 수하물 규정이 업데이트됨. 기존 앱은 단순한 품목명 목록만 보여주고, AI 서버의 Gemini 프롬프트에 규정 컨텍스트가 없어 부정확한 답변 가능성이 있었음.

### 변경 1: BaggageScreen 아코디언 데이터 구조 개선

**Before** — 단순 문자열 배열, 클릭 시 탭별 공통 설명만 표시

```js
items: ['보조배터리 (160Wh 초과)', '라이터 (위탁 불가)', ...]
```

**After** — `{title, detail}` 객체 배열, 클릭 시 품목별 구체적 규정 표시

```js
items: [
  {
    title: '보조배터리 (리튬이온)',
    detail: '위탁 수하물에는 용량 무관 완전 금지입니다. 기내 반입은 100Wh 이하만 허용, 100~160Wh는 항공사 사전 승인 후 최대 2개, 160Wh 초과는 불가합니다.'
  },
  ...
]
```

**추가된 항목**

- 반입금지: 전동킥보드·호버보드, 토치(제트) 라이터 신규 추가
- 제한적반입: 보조배터리 100Wh/160Wh 기준 명시, 노트북, 주류 추가
- 위탁전용: 각 항목별 포장·조건 상세 설명

### 변경 2: AI 서버 Gemini 프롬프트 규정 컨텍스트 주입

**Before** — 규정 컨텍스트 없이 Gemini에 바로 질문

```python
prompt = f"'{item}'을(를) {country} 여행 시 {airline} 항공기에 가져갈 수 있는지 알려줘."
```

**After** — IATA 2024-2025 기준 `REGULATION_CONTEXT` 상수를 프롬프트 앞에 주입

```python
REGULATION_CONTEXT = """
[2024-2025 IATA 항공 수하물 최신 규정 기준]

■ 보조배터리 / 리튬이온배터리
- 위탁 수하물: 완전 금지 (용량 무관)
- 기내 반입: 100Wh 이하 허용 / 100~160Wh 항공사 승인 후 2개 / 160Wh 초과 불가
- 스마트백(배터리 내장 캐리어): 배터리 분리 후 기내 반입, 분리 불가 시 전체 금지

■ 라이터
- 일반 부탄 라이터: 기내 1인 1개 허용, 위탁 완전 금지
- 토치 라이터: 기내·위탁 모두 금지
...
"""

prompt = f"{REGULATION_CONTEXT}\n\n위 규정을 참고해서 '{item}'을(를) ... 판단해줘."
```

주입된 규정 항목: 보조배터리, 액체류(LAGs), 라이터, 칼·가위, 폭발물, 전동킥보드, 스포츠 장비, 주류

### 관련 커밋

- `feat: 2024-2025 IATA 최신 수하물 규정으로 업데이트`

## 7. 커뮤니티 작성자 본인 확인 버그

### 문제

게시글 상세 화면에서 자신이 작성한 글임에도 **수정/삭제 버튼이 나타나지 않음**. 마찬가지로 자신의 댓글에 **삭제 버튼이 표시되지 않음**.

### 원인 분석

`PostDetailScreen.js`에서 본인 여부를 확인하는 로직에 필드 불일치 문제가 두 군데 있었음.

**버그 1 — 게시글 수정/삭제 버튼**

```js
// 버그: userId(로그인 ID)와 authorNickname(닉네임)을 비교
const [myUserId, setMyUserId] = useState(null);
setMyUserId(JSON.parse(stored).userId);          // "jjun012" 같은 로그인 ID
const isAuthor = post.authorNickname === myUserId; // 항상 false
```

`PostResponseDTO`의 작성자 필드는 `authorNickname`(닉네임)인데, AsyncStorage에서 꺼낸 값은 `userId`(로그인 ID)로 두 값이 절대 일치할 수 없음.

**버그 2 — 댓글 삭제 버튼**

```js
// 버그: CommentResponseDTO에 userId 필드가 없음
{c.userId === myUserId && <DeleteButton />}
```

`CommentResponseDTO` 구조:

```java
// id, content, authorNickname, createdAt — userId 필드 없음
```

`c.userId`는 항상 `undefined`이므로 삭제 버튼이 절대 표시되지 않음.

### 해결

AsyncStorage에서 `userId` 대신 `nickname`을 읽어와 `authorNickname`과 비교.

```js
// 수정 후
const [myNickname, setMyNickname] = useState(null);
setMyNickname(JSON.parse(stored).nickname);       // 닉네임으로 변경

const isAuthor = post.authorNickname === myNickname;  // 정상 비교

// 댓글 삭제도 동일하게
{c.authorNickname === myNickname && <DeleteButton />}
```

`UserResponseDTO`에 `nickname` 필드가 있고 로그인 시 AsyncStorage에 저장되므로 항상 사용 가능.

### 관련 커밋

- `fix: 게시글/댓글 작성자 본인 확인 버그 수정`

## 현재 알려진 제한 사항

- `BAGGAGE_RULES`의 초과요금 단위가 USD(`$`)로 표시되나 실제 항공사 요금은 원화(KRW) 기준이므로 추후 수정 필요
- AI 서버 DB lookup(`query_regulation`)의 `regulations` 테이블이 현재 비어 있어 모든 요청이 Gemini fallback으로 처리됨 — 자주 조회되는 품목은 DB에 직접 등록하면 응답 속도 개선 가능
- Supabase username이 `application.yaml`에 하드코딩되어 있음 — 보안상 민감하지 않은 값이나(프로젝트 ref는 공개 정보) 추후 Railway 비밀 변수 처리 고려
